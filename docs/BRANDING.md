# Branding

How the Co-Pilot wears the Gravitas brand. Every visitor-facing artifact — canvas components, the audit report, the executive brief PDF, the daily-cap card, the lead form — carries consistent Gravitas identity.

This doc is the source of truth. Components and report generators read from constants defined here; nothing hard-codes a colour, font, or asset URL elsewhere.

---

## Logo

**Source:** https://assets.thisisgravitas.com/gravitas_logo_large_cf1642efb9.svg

**Rule:** Fetched **once at scaffold time**, saved as a base64-encoded data URI in `src/lib/branding/logo.ts`, and imported as a constant everywhere it's needed. **No remote fetch at runtime.**

Why: a runtime fetch can fail (asset host down, network blocked, PDF generator running in a sandbox). Embedding as a data URI guarantees the logo always renders. The trade-off is a slightly larger bundle and a manual refresh step when the logo changes — both acceptable.

### Scaffold step (Phase 0)

`src/lib/branding/logo.ts` exports two constants:

```ts
// Generated by `pnpm branding:fetch`. Do not edit by hand.
export const GRAVITAS_LOGO_DATA_URI = "data:image/svg+xml;base64,..."; // base64 of the SVG
export const GRAVITAS_LOGO_SOURCE_URL = "https://assets.thisisgravitas.com/gravitas_logo_large_cf1642efb9.svg";
```

A small `pnpm branding:fetch` script (in `scripts/fetch-branding.ts`):

1. Fetches the SVG from `GRAVITAS_LOGO_SOURCE_URL`
2. Validates it's actually SVG (starts with `<svg`)
3. Base64-encodes it
4. Rewrites `src/lib/branding/logo.ts`

Run this once at scaffold and whenever the logo changes. Commit the generated file — it's source of truth.

### Where the logo appears

| Surface | Treatment |
|---|---|
| Canvas pane header | Small mark, top-left of the canvas, fixed position |
| Audit report (canvas + PDF) | Full lockup in the report header |
| Executive brief PDF (Phase 3) | Cover page + footer of every page |
| Daily cap card | Top-centre of the card |
| Lead form | Top-left of the form |
| Loading states | Subtle pulse animation on the mark |

Components import the constant:

```tsx
import { GRAVITAS_LOGO_DATA_URI } from "@/lib/branding/logo";

<img src={GRAVITAS_LOGO_DATA_URI} alt="Gravitas" className="h-6" />
```

In a server-rendered PDF (Phase 3), the same constant is embedded inline.

## Audit report template

Synthesized from the SAB PWS Audit & Research executive summary (April 2026) — a real Gravitas client engagement. **No content from that report is reused.** Only the *structure*, *framing*, and *voice patterns* are codified here. Every agent-generated audit is composed fresh from the visitor's submitted URL.

### Scope discipline — one page per audit

A single agent-generated audit covers **exactly one URL** — the page the visitor submitted. No link-following, no sibling-page crawling, no sitemap walking. This:

- Bounds compute and Anthropic spend predictably (see `ARCHITECTURE.md` → Cost cap)
- Respects the per-IP `IP_DAILY_AUDIT_LIMIT=1` rate limit
- Keeps the report focused; a multi-page audit done in 60s is a worse output than a single-page audit done well
- Honestly distinguishes the auto-audit from a full Gravitas engagement (which audits 30+ pages)

If the visitor wants a different page audited, they wait for the next reset (00:00 UTC) and submit again. The `RateLimitReached` card explains this politely.

### Honest scaling — what the auto-audit DOES NOT do

The agent-generated audit MUST be transparent about the gap between itself and a real Gravitas engagement. The report includes a short "About this audit" footer block:

> *This is a 60-second auto-audit of one page. A full Gravitas engagement audits 30+ pages across personas, benchmarks against direct competitors, and produces a multi-month transformation roadmap. What you're seeing here is the starting point of that conversation.*

This framing positions the deeper engagement instead of pretending to replace it. **Never** generate fake competitor scores, fake persona-testing data, or fake "X banks benchmarked" stats. If we don't have the data, we say so.

### The Four-Lens Framework

Inherited from the Gravitas methodology, used in every audit:

| Lens | Question | Max | What we measure (single-page) |
|---|---|---|---|
| **D1 — Usability Standards** | Is it usable? | /30 | Nielsen heuristics applied to the single page: visibility of system status, consistency, error prevention, recognition over recall, flexibility, aesthetic + minimalist design, help users recognise/diagnose/recover from errors. 10 heuristics × 0–3. |
| **D2 — User Needs** | Can users achieve their goals? | /20 | **Framework-based only**, not user-tested. Inferred from page structure: is the user's likely goal addressable (Discover → Evaluate → Act)? Information needed to decide present? Path-to-action clear? Explicitly labelled "framework-based" in the report. |
| **D3 — Conversion** | Does it convert visitors into customers? | /30 | CTA architecture, decision-supporting content, trust signals, funnel completeness, fee/pricing transparency, above-fold action availability. |
| **D4 — Design Execution** | How well is it built? | /20 | Component coherence, accessibility (contrast, touch targets, alt text, ARIA, keyboard nav), mobile experience, typographic consistency, RTL/i18n handling if applicable, page performance (LCP/CLS/TTFB from Lighthouse). |

Total: **/100**, with the post-engagement target offered as the conversion narrative ("a typical post-engagement target is 75/100").

| 6 | **D1 Deep-dive** | findings tagged `usability` | `AuditFindings` filtered by lens | Themes (3–4) + findings list + visual callouts |
| 7 | **D2 Deep-dive** | framework inference | `AuditFindings` filtered by lens | Discover / Evaluate / Act framing with page-specific commentary |

### Voice rules — phrases the agent uses

Adapt these patterns directly. Always replace `[domain]` with the visitor's domain.

- **Section opener:** *"[domain] today."* — present-tense, declarative, neutral
- **Sub-opener for lens deep-dives:** *"D{N} — How well does the website [do X]?"*
- **Strength framing (Keep & Build On):** *"This is a genuine [competitive strength / starting point]. Use this as the internal benchmark."*
- **Lesson callout (after a competitive pattern):** *"Lesson: [single sentence imperative]."*
- **Forward-looking close on a lens:** *"[Subject] sits at [position] today — a strategic position that, with the right investment now, leapfrogs the competition rather than catches up to it."* Use sparingly; once per report is enough.
- **About-this-audit caveat:** the full block quoted in "Honest scaling" above. Word-for-word.
- **Closing:** *"Shukran! / شكراً"* — always bilingual gesture

### Voice anti-patterns — never use

- *"Cutting-edge"*, *"leverage"*, *"best-in-class"* (Gravitas voice rules)
- Hedging: *"you might want to consider possibly..."* — be direct
- Fake stats: *"X% of users..."* without source — never
- Pretending the auto-audit is a full engagement
- ChatGPT-style listicles with 10+ items per section (3–6 max per group)

### Scoring rubric — calibration

Every lens score the agent emits has a calibrated meaning. Document at top of every audit report (small print, glossary-style):

| Range | D1 / D3 (out of 30) | D2 / D4 (out of 20) | Meaning |
|---|---|---|---|
| Excellent | 24–30 | 16–20 | Category-leading on this dimension |
| Strong | 18–23 | 12–15 | Above market average, room to widen the gap |
| Developing | 12–17 | 8–11 | At market average; the lens that moves the audit score most |
| Weak | 0–11 | 0–7 | Material risk to conversion or brand |

Be conservative. **Most real pages score Developing**. Inflating to Strong is worse than being honest — Gravitas-quality audits earn trust by being calibrated, not flattering.

### Findings convention

Each finding emitted into `AuditFindings` follows the same shape:

- **Title** (≤ 10 words): the issue, not the fix. *"Apply CTA below the fold on mobile"*, not *"Move Apply CTA above the fold"*
- **Detail** (≤ 60 words): why it matters, what the visitor sees, what the impact is. No code snippets in Phase 1.
- **Severity** (`critical` / `high` / `medium` / `low`): based on user-impact, not technical complexity
- **gravitasService** tag: maps to one of the five services (see `VISION.md`)
- The fix lives in the corresponding Roadmap item, not the finding itself

### Visual evidence (Phase 3)

The sample report's most powerful element is the *Subject vs Competitor* side-by-side screenshot pair. For Phase 3 PDF generation:

- A single annotated screenshot of the audited page, with callouts pointing at top findings
- No fake competitor screenshot — instead, a stylized "what good looks like" diagram or omit the right column entirely
- Screenshot captured by the crawl worker (`worker/src/screenshot.ts`) at standard mobile + desktop viewports

Phase 1 canvas does NOT include the screenshot — it would bloat the experience and Lighthouse/Playwright screenshots are heavy. Add when PDF generation lands.

### Closing contact — rotation

The "Shukran!" page in a real Gravitas report names a real person. For the auto-audit:

- A `BRANDING_CLOSING_CONTACT` config exposes `name`, `role`, `phone`, `email`, `headshot_data_uri`
- One source of truth, env-overridable per-deployment
- Default: TBD — Nikki to provide the contact for the production deployment
- Never let the LLM invent a name or contact details
